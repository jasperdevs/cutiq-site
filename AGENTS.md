# Global Rules

## Workflow
- Break work into small steps. Verify each step before moving on (test, typecheck, build).
- When the task is ambiguous, ask. Do not assume scope, design, or requirements.
- Search for line numbers first, then read targeted ranges. Never read entire large files.
- Batch edits across files, then run one build. Do not rebuild per edit.
- Capture command output once. Do not re-run the same command to re-analyze.
- IMPORTANT: Never speculate about code you have not read. Read the relevant files before answering or making changes.

## Code
- Extract shared logic into reusable modules. Do not duplicate a pattern that exists or could exist as a single source of truth.
- Prefer editing existing files over creating new ones.
- Never remove or change code unrelated to the current task.
- Delete dead code. Never comment it out.
- Match existing codebase style. Do not impose a different style.
- Avoid `any` in TypeScript. Write explicit types when not obvious from context.
- Do not overengineer. Only make changes that are directly requested or clearly necessary. Do not add abstractions, helpers, or features beyond what was asked.

## Git
- IMPORTANT: Always commit and push after completing work.
- Use conventional commits (`feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `test:`). Subject under 72 chars.
- Never add AI attribution, co-author lines, or "Generated by" footers.

## Safety
- Never commit `.env`, API keys, secrets, or credentials.
- If a secret is missing, stop and ask. Do not invent placeholders.

## Communication
- Be direct. State what you found, what you did, what remains.
- Say "I don't know" when uncertain. Do not guess.
- Reference code as `path/file.ts:42`. Do not generate docs unless asked.
